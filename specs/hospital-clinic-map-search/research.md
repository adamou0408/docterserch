## 可行性評估

### 技術風險
| 風險項目 | 風險等級 | 說明 |
|----------|----------|------|
| 地圖服務整合 | 中 | 需整合 Google Maps API 或 Leaflet/OpenStreetMap，涉及 API key 管理與費用 |
| 醫院門診資料來源 | 高 | 台灣各醫院門診時間無統一 API，需爬取官網或整合健保署開放資料，資料格式不一致 |
| 資料即時性 | 中 | 門診時間可能隨時異動，需定期更新機制（ETL pipeline）|
| 掛號系統整合 | 高 | 各醫院掛號系統獨立，直接掛號需要與各醫院系統對接，初期可能僅提供連結導向 |
| 資料量與效能 | 低 | 台灣醫療院所數量有限（~2萬），搜尋範圍以地理位置篩選後資料量可控 |

### 是否涉及以下高風險領域？
- [x] 資料庫 schema 變更（需要遷移策略）— 需建立醫院、科別、醫生、門診時段等資料表
- [x] 新的外部依賴或第三方整合 — 地圖 API、健保署開放資料、各醫院官網資料
- [ ] 安全敏感操作（認證、授權、加密）— 初期為公開查詢，不涉及個資
- [x] 效能敏感操作（大量資料處理、即時運算）— 地理位置搜尋需要空間索引（PostGIS 或 Elasticsearch geo query）
- [x] 基礎設施變更（新服務、新環境）— 需要資料庫、資料爬取排程、地圖前端元件

## 建議方向

### 核心架構建議
1. **資料層**：優先整合台灣健保署開放資料（醫事機構基本資料）作為醫院清單基礎，再逐步爬取各醫院門診時刻表
2. **地圖元件**：建議使用 Leaflet + OpenStreetMap（免費）作為 MVP，後續可視需求升級至 Google Maps
3. **搜尋機制**：以使用者選定位置為中心，設定半徑範圍（如 3km/5km/10km），篩選指定科別的醫療院所
4. **門診資料更新**：建立定時爬蟲（每日/每週）更新各醫院門診時刻表，存入結構化資料庫

### 建議開發分期
- **Phase 1 (MVP)**：地圖顯示 + 健保署資料匯入 + 基本科別篩選
- **Phase 2**：門診時刻表爬取 + 醫生資訊顯示
- **Phase 3**：掛號連結導向 + 即時掛號狀態查詢（視各醫院 API 開放程度）

### 關鍵資料來源
- 健保署醫事機構基本資料開放資料集（含地址、科別、經緯度）
- 各醫院官方網站門診時刻表頁面
