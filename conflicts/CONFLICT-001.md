# 衝突編號：CONFLICT-001

## 狀態：`resolved`

## 衝突描述
- **角色 A（患者）** 需要：門診資訊即時正確，包括「目前是否仍可掛號」的即時狀態（US-3），以及能在搜尋結果中依「最近可掛號時間」排序（US-4）。患者期望看到的資訊與醫院官網完全一致，避免白跑一趟。
- **角色 B（系統管理員）** 需要：資料爬取頻率需在成本與技術可行性之間取得平衡（US-5）。各醫院官網格式不一、可能隨時改版導致爬蟲失效，資料覆蓋率難以保證 100%。
- **為什麼衝突**：患者期望的「即時掛號狀態」需要接近即時的資料同步（分鐘級），但爬蟲架構受限於來源網站的爬取頻率限制、格式解析穩定性、以及運算成本，實務上只能達到每日或每數小時更新一次。這導致資料可能存在延遲，使用者看到「可掛號」但實際已額滿的情況。

## AI 分析

這是一個典型的 **資料新鮮度 vs 系統成本** 的 UX 衝突。患者的核心痛點是「不想白跑一趟」，而系統的限制是「無法保證即時同步」。此衝突的嚴重程度為**中等**——不會阻擋功能上線，但若處理不當會嚴重影響使用者信任度。

## 解決方案
- [ ] 方案 1：（優先滿足患者信任感）在門診資訊旁明確標示「資料更新時間」，讓患者自行判斷資訊新鮮度。對於超過特定時間未更新的資料，顯示警告提示「此資訊可能已變動，建議電話確認」。
- [ ] 方案 2：（優先滿足系統穩定性）初期不顯示「是否可掛號」的即時狀態，僅顯示固定的門診時刻表（每週哪天有哪些醫生看診）。將 US-3 中的即時掛號狀態標記為 Phase 2 功能。
- [ ] 方案 3：（折衷方案）對有提供 API 或結構化資料的醫院顯示較即時的掛號狀態，對僅能爬取的醫院則顯示門診時刻表並附上「資料更新時間」標記。依資料來源品質分級顯示不同的信心指標。

## 決策紀錄
- **決策者**：產品經理/開發者
- **決策日期**：2026-04-06
- **選擇方案**：方案 3（依資料品質分級）
- **理由**：兼顧使用者體驗與系統可行性。有 API 的醫院提供即時狀態，僅能爬取的醫院則顯示時刻表加更新時間標記，讓使用者依信心指標自行判斷。
